# Sprint Document Isolation — Design

## Problem

When using Harness scaffold files, new sprints keep modifying existing docs from previous sprints instead of creating new documents. Root causes:

1. Artifact paths in `stageArtifacts.ts` are static — every sprint points to the same files (e.g., `docs/design/problem-statement.md`)
2. Commands instruct AI to "create or update" / "append" instead of "create new"
3. Archive logic in `session-init.sh` only triggers when `started_at` is set — interrupted sprints leave old files behind
4. Archive matches files from `constraints.json` gate patterns, which may not cover all artifacts

## Solution: Sprint Number Tracking + Robust Archiving

### Section 1: Sprint Number Tracking

Add `sprint.number` to `state.json`:

```json
{
  "sprint": {
    "current": "think",
    "number": 3,
    "history": ["think", "plan"],
    "started_at": "2026-04-13T10:00:00Z"
  }
}
```

- Integer counter, incremented on each `session-init.sh` run
- Starts at 1 if no prior state.json exists
- If state.json exists (regardless of sprint completion), read old number + 1

**Archive suffix format:** `{name}.{number}.{date}.{ext}`
- Example: `problem-statement.001.2026-04-10.md`, `architecture.002.2026-04-12.md`

**Trigger condition:** Archive whenever state.json exists + artifact files exist. No dependency on `started_at`.

### Section 2: Archive Source — stage-artifacts.json

Change archive matching source from `constraints.json` gate patterns to `.harness/stage-artifacts.json` (already generated by `flows.ts`). This file contains the complete artifact manifest:

```json
{
  "think": [
    {"path": "docs/design/problem-statement.md", "verification": "contains-section", "sectionMarker": "## Problem Statement"},
    {"path": "docs/design/scope.md", ...}
  ],
  "plan": [...]
}
```

**Updated archive flow (session-init.sh):**

1. Check if `state.json` exists → if yes, trigger archive
2. Read `sprint.number` and `sprint.started_at` (for date) from state.json
3. Read `stage-artifacts.json`, extract all unique paths
4. For each existing artifact file, rename to `{name}.{number}.{date}.{ext}`
5. Handle same-name collisions (add counter for multiple inits on same day)
6. Delete state.json → triggers fresh initialization below (number auto-increments)

### Section 3: Command Behavior — Create New

Change all artifact output instructions in generated commands from "create or update/append" to "create new file".

Specific changes in `commands.ts`:
- `generateBuildProcess()` — build report instruction
- `STATIC_COMMANDS.review.process` — review report instruction
- `STATIC_COMMANDS.test.process` — test report instruction
- `STATIC_COMMANDS.ship.process` — release notes instruction
- `STATIC_COMMANDS.reflect.process` — retrospective report instruction

New wording: "Create a **new** file. If the file already exists, it means sprint init archiving was not run — run `bash .claude/scripts/session-init.sh` first and retry."

**Exception:** `generateNewTaskCommand()` Category B keeps append behavior (intra-sprint task addition, not new sprint).

### Section 4: Files Changed

| File | Change |
|------|--------|
| `src/generators/engines/claude/sessionScripts.ts` | Rewrite archive logic: use `stage-artifacts.json`, add sprint number, remove `started_at` dependency |
| `src/generators/engines/claude/commands.ts` | Change all artifact instructions from "create or update/append" to "create new" |

No changes needed to:
- `stageArtifacts.ts` — artifact paths remain static
- `flows.ts` — flow files unchanged
- `hooks.ts` — gate checks static paths; after archiving, old files are moved away and new files don't exist yet, so gates naturally start as pending

### Data Flow

```
User runs session-init.sh
  → Read stage-artifacts.json for all artifact paths
  → Archive existing files (add {number}.{date} suffix)
  → Delete old state.json
  → Create new state.json (number+1, all gates pending)
  → AI runs /think → creates new problem-statement.md (fresh file)
  → Gate check passes → continue
```
